1. Core Data Model

Every feature in AyuAware is built on two data layers per user. Understanding the distinction between them is prerequisite to understanding any flow.

Static Layer — Prakriti (Constitution)

Table: user_constitution  |  Set: once at onboarding  |  Mutable: no

Prakriti is the user's birth constitution in Ayurvedic medicine. It is immutable — determined at conception and stable throughout life. It is the baseline against which all imbalances are measured.

FieldTypeNotes
vata_percentageinteger 0–100Vata dosha share. Fixed at assessment. vata+pitta+kapha always = 100.
pitta_percentageinteger 0–100
kapha_percentageinteger 0–100

assessment_responsesJSONBRaw 60-question answers stored for re-scoring and auditability
assessed_attimestamptzUsers may re-assess after 90 days

Dynamic Layer — Vikriti (Current State)

Table: user_current_state  |  Updated by: check-ins, wearable sync, lab upload, scan analysis, manual logs

Vikriti represents the user's current imbalance state, expressed as deltas from the Prakriti baseline — not as absolute percentages. A vata_imbalance of +3 means Vata is currently aggravated by 3 units above that user's natural Vata constitution.

Vikriti imbalances do NOT replace or overwrite Prakriti percentages. They are additive deltas. The AI always reads both: "Your constitution is 65% Vata (your nature), currently showing +3 Vata aggravation (your state right now)."

2. Interaction Flows

Seven backend flows describe every way the system processes a user action or background event. Each flow specifies: trigger, ordered steps with actor labels, DB reads/writes, and AI calls.

Flow 01 — Onboarding

Purpose: Capture Prakriti and store it as the user's permanent constitutional baseline. Generate first protocol. Runs: once per user.

  1. User Registers (email / Apple Sign-In / Google)
  2. User Answers 60-question Prakriti assessment (5 sections: physical, physiological, mental/emotional, lifestyle, current state)
  3. Backend Scoring algorithm runs server-side → vata%, pitta%, kapha% (always sum to 100)
  4. Backend INSERT user_constitution with percentages and raw responses
  5. Backend INSERT user_current_state with defaults: all imbalances = 0, agni = sama, ama = none, ojas = 7
  6. User Names Digital Twin, selects communication style
  7. iOS (Optional) Authorises Apple HealthKit → backend enqueues wearable_initial_sync job (reads last 30 days)
  8. AI Twin Deep-dive onboarding conversation (conversation_type = 'onboarding') — populates user_health_profile from conversation
  9. AI Twin Generates first protocol via create_protocol tool. Herb safety check against user medications runs automatically.
  10. User Reviews protocol draft and accepts
  11. Backend UPDATE protocols.status = 'active'. SET users.onboarding_complete = true.
DB reads
users, user_health_profile
DB writes
users, user_constitution, user_current_state, user_health_profile, user_twin_preferences, user_wearable_devices, protocols, protocol_versions
AI calls

Flow 02 — Daily Check-In

Purpose: Primary mechanism for updating Vikriti. 5-question form.

  1. User Submits: energy_level (1–10), sleep_quality (1–10), digestion_state, mood, symptoms[]
  2. Backend INSERT daily_checkins (TimescaleDB)
  3. Backend Apply Vikriti update rules (see table below)
  4. Backend UPDATE user_current_state with new deltas
  5. Worker Async: check alert triggers → if same signal 3+ days → enqueue prediction_alert
  6. AI (Haiku) Async: generate 2–3 sentence insight + 1 tip → save as message

Flow 03 — Chat Query

Purpose: Core product interaction. User message → context assembled → Claude streams → saved.

Transport: WebSocket (Socket.io). Events: chat:send (client→server), chat:chunk (server→client, streaming), chat:complete, chat:error.

  1. Backend Auth check + usage limit check (Free: 20 msgs/month)
  2. Backend Context assembly — priority order: Prakriti → Vikriti → active protocol → last 7 check-ins → conversation history (trimmed first if over budget) → date/season
  3. Backend Claude API call: model by tier, stream: true, tools: AYUAWARE_TOOLS[10]
  4. Backend Stream loop: content_block_delta → forward chunk to client. tool_use → PAUSE forwarding, execute tool, return result to Claude, RESUME.
  5. Backend On stream end: save conversation + both messages + metadata. INCREMENT usage_counters.

Flow 04 — Wearable Sync (Apple HealthKit)

Purpose: Passive background flow. Device data → Ayurvedic interpretation → Vikriti update.

Sync cadence: foreground open → last 24h. Hourly background fetch → last 1h. Initial connect → last 30 days.

  1. iOS Query HealthKit: SleepAnalysis, HRV, HeartRate, StepCount, ActiveEnergyBurned since last_synced_at
  2. Backend BATCH INSERT wearable_data_points (TimescaleDB)
  3. Backend Apply interpretation rules (see table)
  4. Backend If signals consistent 3+ days: UPDATE user_current_state
  5. Backend UPDATE user_wearable_devices.last_synced_at
  6. Worker Check: HRV declining >20% over 3 days → enqueue prediction_alert

Flow 05 — Lab Upload Premium only

Purpose: PDF lab results → OCR → biomarker extraction → Ayurvedic interpretation → Vikriti update.

  1. Backend Issue presigned S3 URL
  2. iOS Upload PDF directly to S3
  3. Backend INSERT lab_results { status: 'pending' }. Enqueue lab_ocr_processing job.
  4. Worker Download PDF. Call Claude Vision to extract all biomarkers (name, value, unit, reference range).
  5. Worker Map each biomarker → dhatu (tissue layer) + dosha. Flag out-of-range values. Escalate critically abnormal as medical_flags.
  6. Worker Generate overall_pattern, root_cause_analysis, action_plan[].
  7. Worker UPDATE lab_results { status: 'complete' }. Send push notification. Reassess Vikriti.

Flow 06 — Protocol Creation

Purpose: AI generates a structured 3-phase wellness plan. Herb safety check is mandatory.

Triggers: onboarding completion | user chat message | direct UI action

  1. Backend Assemble context: Prakriti + Vikriti + active symptoms + health goals + lifestyle constraints + current season + existing protocols
  2. AI (Sonnet) Generate protocol: 3 phases (Foundation → Building → Maintenance), 5 categories (Dietary / Lifestyle / Herbs / Practices / Environmental). Each recommendation: action, frequency, timing, rationale, tips, contraindications, alternatives.
  3. Backend Herb safety check: cross-reference all herb recommendations against user medications and conditions. Remove or flag unsafe items.
  4. Backend INSERT protocols { status: 'draft' }. INSERT protocol_versions { v1 }.
  5. User Reviews protocol card. Accepts.
  6. Backend UPDATE protocols.status = 'active'. Enforce plan limits (Free/Personal: max 1 active). Schedule reminders.
Plan limits
Free: 1 active · Personal: 1 active · Premium: 3 active
DB writes
protocols, protocol_versions, notifications
AI calls
claude-sonnet-4-5 — max_tokens: 4096 — tools: create_protocol, check_herb_safety

3. AI Tool Definitions

10 tools are available to Claude during chat. The backend's ToolExecutorService executes each tool; Claude does not call external services directly.

Execution model: Claude emits tool_use block → backend pauses stream forwarding → executes NestJS service method → returns tool_result to Claude → stream resumes.

Important: Herb and food knowledge (lookup_herb, lookup_food, check_herb_safety) is generated from Claude's training data, not from a pre-seeded practitioner-validated database. Results are cached in herbs / foods tables with ai_generated = true until reviewed by a qualified practitioner.

detect_emergencysafetyCalled first if message contains emergency keywords. Overrides all normal flow. Returns emergency directive template.audit_logs
check_herb_safetysafetyReads user medications/conditions from DB. Generates safety assessment from Claude's herb knowledge. Always call before recommending a herb.herbs (cache)
create_protocolprotocolGenerates and saves a new wellness protocol draft.protocols, protocol_versions
modify_protocolprotocolAdjusts an existing active protocol. Snapshots before modifying.protocol_versions, protocols
get_active_protocolsprotocolReturns user's current active protocols with today's recommendations.—
log_checkindataLogs a check-in when user mentions their state in conversation (not via form).daily_checkins, user_current_state
log_manual_entrydataLogs meals, symptoms, supplements, meditation etc. mentioned in conversation.manual_logs
get_user_health_summarydataFetches comprehensive current health state for mid-conversation context refresh.—
