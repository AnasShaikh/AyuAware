⚙️ Tech Stack (click to expand)

📱 Mobile App Stack (Frontend)

The technologies that will build the actual AyuAware app on the user's phone.

⚛️ React Native (with TypeScript)

Role in AyuAware: The core framework building the iOS (and later Android) application.

This is what actually renders the buttons, screens, and text on the phone. By choosing React Native, we write one single codebase that works natively on both iOS and Android.

Why for MVP: It is highly "agent-friendly." Because it's one of the most popular frameworks in the world, AI coding assistants have massive amounts of training data on it, enabling the generation of reliable, bug-free UI code much faster than using pure Swift (iOS) or Kotlin (Android).

🛠️ Expo

Role in AyuAware: The build system and toolkit that makes React Native easy to manage.

Normally, building a mobile app requires fighting with Apple's Xcode or Android Studio. Expo handles all of that complex compiling in the cloud.

Why for MVP: It allows us to preview the app instantly on physical phones via a QR code. More importantly, it allows for Over-The-Air (OTA) updates. If we find a bug in the protocol UI, we can push a fix directly to users' phones immediately, without waiting for Apple's App Store review process.

⚙️ Backend Stack & Infra (MVP)

The servers, databases, and APIs that power the AyuAware intelligence.

🟢 Node.js API (with TypeScript)

Role in AyuAware: The "Brain" of the system.

This is the server code that receives data from the mobile app, talks to the AI (OpenAI/Gemini), talks to Spike (wearables), calculates logic, and sends responses back.

Why for MVP: It uses the exact same language (TypeScript) as the mobile app. This means data models can be seamlessly shared between the app and the server without translation errors, significantly accelerating AI-assisted development.

🐘 PostgreSQL Database (Neon)

Role in AyuAware: The permanent memory storage.

This holds all user profiles, check-ins, health data, and generated protocols. It enforces strict rules so data doesn't get corrupted.

Why for MVP: Neon gives managed serverless Postgres with strong reliability and fast setup while keeping full SQL/Postgres compatibility.

Plan path: Free: $0 in development → Launch: usage-based (typical ~$15/month) at launch → scale to higher usage tiers as growth increases. (Pricing can change; verify before go-live.)

◭ Prisma (ORM)

Role in AyuAware: The safe translator between the Node.js API and the Database.

Instead of writing raw, easily-broken SQL queries to fetch user data, Prisma lets the code interact with the database using safe, autocomplete-friendly commands.

Why for MVP: It provides strict data typing. If a database query attempts to access a field that doesn't exist, Prisma catches the error *before* the code even runs, preventing app crashes. This strict typing is critical for rapid, AI-assisted code generation.

🧠 LLM & Intelligence Layer

Role in AyuAware: The logic engine driving the Digital Twin, diagnosis, and protocol generation.

Default model for MVP: Claude Sonnet 4.6. We start with Claude for strong instruction-following, consistent structured outputs, and high-quality conversational behavior for frequent user interactions.

Provider flexibility (built-in): the backend uses a unified routing layer (e.g., Vercel AI SDK or LiteLLM) instead of hardcoding one vendor. This keeps model/provider switching simple as we add OpenAI, Gemini, and others.

Future intelligence layer evolution: this layer will expand with a vector database for key long-term context (conversation memory, important user patterns) and a fine-tuned model trained on user interactions plus real successful-case outcomes.

☁️ External Services (incl. AWS S3 or Cloudflare R2)

Role in AyuAware: Plugging in existing tools to save time.
  • Better Auth / NextAuth: Handles secure user login (Apple/Google/Email) without us building security from scratch.
  • Spike API: Ingests Apple Health/Oura data cleanly without us writing custom parsers for every watch brand.
  • AWS S3 (or S3-compatible storage like R2): Stores user-uploaded tongue scans and lab report PDFs.

Why not store files directly in Neon? We keep binary files (images/PDFs) in object storage because it is cheaper and optimized for large files, while Neon/Postgres stays focused on structured app data and file metadata (URLs, timestamps, analysis status). This keeps database performance healthy as usage grows.

🧠 Intelligence Prompt Architecture (click to expand)

Defines how AI behavior is controlled consistently across all screens.

Core Model

Every AI request uses a two-layer prompt system:

  • Layer 1: Master Prompt — global behavior, safety, tone, Ayurvedic rules, and medical boundaries.
  • Layer 2: Screen Sub-Prompt — screen/action-specific instructions (e.g., NSL plan, exposure analysis, astrology insight, protocol update).

Base Context Assembly Contract (Canonical)

This is the mandatory context bundle sent on every AI call before any screen-specific additions.

prakritiuser_constitutionLatest assessed constitution

wearables_snapshotwearable_data_pointsLast 72 hours
recent_chat_summaryconversations/messagesRolling summary of last 20 messages (or 7 days)
environment_exposureexposure_impact_eventsLast 48 hours

Conflict Resolution & Precedence

  • Wearable data wins whenever wearable data conflicts with manual or inferred values (for measurable health signals).
  • Manual user input is second priority and can be used when wearable data is missing or clearly invalid.
  • Model-inferred values are advisory only and never override explicit user input.

Missing Context Behavior

  • If core context is missing (user_profile, prakriti, or latest_vikriti): block and ask user.
  • If secondary context is missing (wearables, exposure, recent chat summary): continue with partial context and reduced confidence.
  • All partial-context responses must set status="fallback" or meta.confidence below normal threshold.

Confidence Thresholds

  • confidence < 0.50: fallback response only, no strong recommendations.
  • 0.50 - 0.70: cautious recommendations with explicit uncertainty wording.
  • > 0.70: standard response path.

User-Facing Safety Fallback Copy (v1)

  • MODEL_TIMEOUT: "I could not complete this analysis in time. Please retry in a moment."
  • INVALID_OUTPUT: "I could not generate a reliable recommendation right now. Please try again."
  • SAFETY_BLOCK: "I cannot safely answer this directly. Please consult a qualified practitioner for this decision."
  • MISSING_CORE_CONTEXT: "I need a few required details before I can generate an accurate recommendation."

Fallback Rule (Mandatory)

  • If sub_prompt_id exists: run Master Prompt + Sub-Prompt.
  • If no sub-prompt exists: run Master Prompt only (safe default behavior).
  • If output fails schema validation: return safe fallback response, log error, and avoid unsafe/partial guidance.

One-Call Carousel / Sub-screen Rule

  • All visible and latent carousel panel data must be returned in the same response.
  • Frontend may reveal panels progressively on slide, but no second AI call is required for panel content.
  • Heavy assets can be referenced as URLs in asset_refs, but text/insight payloads must be preloaded in subscreens[].
